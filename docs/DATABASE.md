# Database Documentation

## Overview

Gorgonetics uses two database layers, each suited to its workload:

- **Auth database** (`users.sqlite`): A plain SQLite file for users and sessions. Provides proper ACID transactions for authentication without DuckLake snapshot bloat.
- **DuckLake** (`metadata.sqlite` + `data/*.parquet`): A versioned, snapshot-based analytical store for gene and pet data. The catalog (schema metadata) lives in a SQLite file, while actual table data is stored as Parquet files.

No separate database server is required.

## Architecture

```
project root/
  users.sqlite             # Auth database -- users, sessions (SQLite)
  metadata.sqlite          # DuckLake catalog -- schema, snapshot metadata
  data/                    # Parquet files -- gene and pet data
    <auto-generated>.parquet
```

- **Auth database**: `users.sqlite` (configurable via `GORGONETICS_AUTH_DB_PATH`)
- **DuckLake catalog**: `metadata.sqlite` (configurable via `GORGONETICS_CATALOG_PATH`)
- **Data directory**: `data/` (Parquet files managed by DuckLake)
- **DuckLake name**: `gorgonetics_lake` (the attachment alias used inside DuckDB)

The `AuthDatabase` class in `src/gorgonetics/auth/database.py` manages user/session operations. The `DuckLakeGeneDatabase` class in `src/gorgonetics/ducklake_database.py` manages gene and pet data. Instances are created through the factory functions `create_auth_database_instance()` and `create_database_instance()` in `src/gorgonetics/database_config.py`.

### Connection flow

1. `create_database_instance()` reads `DatabaseConfig.from_environment()`.
2. A `DuckLakeGeneDatabase` is created: DuckDB connects in-memory, installs and loads the `ducklake` and `sqlite` extensions, then attaches the DuckLake catalog.
3. Tables are created if they do not already exist.
4. Demo pets are loaded for anonymous browsing (user_id = -1).

## Configuration

### Environment variables

| Variable | Default | Description |
|----------|---------|-------------|
| `GORGONETICS_AUTH_DB_PATH` | `users.sqlite` | Path to the SQLite auth database |
| `GORGONETICS_CATALOG_PATH` | `metadata.sqlite` | Path to the DuckLake SQLite catalog |
| `GORGONETICS_DATA_PATH` | `data` | Directory for Parquet data files |
| `GORGONETICS_DUCKLAKE_NAME` | `gorgonetics_lake` | DuckLake attachment name |
| `GORGONETICS_CONCURRENT_ACCESS` | `false` | Enable concurrent access mode |
| `GORGONETICS_SESSION_TIMEOUT` | `30` | Session timeout in minutes |
| `DUCKDB_MEMORY_LIMIT` | *(none)* | DuckDB memory cap (e.g. `256MB`). Set on constrained VMs. |

You can place these in a `.env` file in the project root (python-dotenv is loaded automatically if installed).

## Schema

### Auth database (`users.sqlite`)

#### users

Stores registered accounts. Passwords are bcrypt-hashed. Registration is invite-only (admin creates accounts).

```sql
CREATE TABLE IF NOT EXISTS users (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    username        TEXT         NOT NULL UNIQUE,
    password_hash   TEXT         NOT NULL,
    role            TEXT         NOT NULL DEFAULT 'user',  -- 'admin' or 'user'
    is_active       INTEGER      NOT NULL DEFAULT 1,
    created_at      TEXT         NOT NULL,
    updated_at      TEXT         NOT NULL
);
```

#### user_sessions

Tracks JWT refresh tokens for session management.

```sql
CREATE TABLE IF NOT EXISTS user_sessions (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     INTEGER      NOT NULL,
    token_jti   TEXT         NOT NULL,
    expires_at  TEXT         NOT NULL,
    created_at  TEXT         NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

### DuckLake (`metadata.sqlite` + `data/`)

#### genes

Static reference data for genetic effects, populated from JSON asset files.

```sql
CREATE TABLE IF NOT EXISTS genes (
    animal_type       VARCHAR NOT NULL,
    chromosome        VARCHAR NOT NULL,
    gene              VARCHAR NOT NULL,
    effectDominant    VARCHAR DEFAULT 'None',
    effectRecessive   VARCHAR DEFAULT 'None',
    appearance        VARCHAR DEFAULT 'None',
    notes             VARCHAR DEFAULT 'None',
    created_at        TIMESTAMP,
    updated_at        TIMESTAMP,
    last_modified_by  INTEGER,
    last_modified_at  TIMESTAMP
);
```

### pets

Dynamic columns are generated at startup from `AttributeConfig`. The base columns are always present; species-specific attribute columns (e.g., `ferocity` for beewasps, `temperament` for horses) plus the core attribute columns (`intelligence`, `toughness`, `friendliness`, `ruggedness`, `enthusiasm`, `virility`) are appended automatically.

```sql
CREATE TABLE pets (
    id              INTEGER,
    user_id         INTEGER      NOT NULL,
    name            VARCHAR      NOT NULL,
    species         VARCHAR      NOT NULL,
    gender          VARCHAR      NOT NULL,
    breed           VARCHAR,
    breeder         VARCHAR,
    content_hash    VARCHAR      NOT NULL,
    genome_data     JSON         NOT NULL,
    notes           TEXT,
    is_public       BOOLEAN,
    created_at      TIMESTAMP,
    updated_at      TIMESTAMP,
    -- dynamic attribute columns follow (e.g., intelligence, toughness, ...)
);
```

Key fields:
- `user_id`: Owner of the pet (-1 for demo pets).
- `content_hash`: SHA-256 of the original genome file, used for duplicate detection.
- `genome_data`: Full genome stored as JSON.
- `is_public`: Whether the pet is visible to other users.

## CLI Commands

All commands are run through the Typer-based CLI.

```bash
# Populate the database with gene data from assets/beewasp/ and assets/horse/
uv run gorgonetics populate

# Show database configuration and connection status
uv run gorgonetics db-status

# List all DuckLake snapshots (version history)
uv run gorgonetics db-snapshots

# Clean up old DuckLake parquet files (dry run by default)
uv run gorgonetics db-cleanup
uv run gorgonetics db-cleanup --no-dry-run   # actually delete files

# User management
uv run gorgonetics create-admin --username admin --password yourpassword
uv run gorgonetics list-users
uv run gorgonetics set-role --username someone --role admin
uv run gorgonetics delete-user --username someone

# Clean up test artifacts
uv run gorgonetics clean-test-artifacts
```

## Data Population

Gene data lives in JSON files under `assets/`:

```
assets/
  beewasp/
    beewasp_genes_chr01.json
    beewasp_genes_chr02.json
    ...
  horse/
    horse_genes_chr01.json
    horse_genes_chr02.json
    ...
```

Each file contains an array of gene objects:

```json
[
  {
    "gene": "01A1",
    "effectDominant": "Intelligence+",
    "effectRecessive": "Intelligence-",
    "appearance": "Brighter antenna glow",
    "notes": "Commonly observed trait"
  }
]
```

Running `uv run gorgonetics populate` clears existing gene rows, loads every `*_genes_chr*.json` file, and commits a DuckLake snapshot.

The web server can also auto-populate an empty database on startup when `GORGONETICS_LOAD_SAMPLE_DATA=true` is set.

## Snapshots and Versioning

DuckLake creates a snapshot on every `conn.commit()`. Snapshots provide a built-in audit trail: you can list them, compare table changes between any two snapshots, and roll back if needed.

```bash
# View snapshots via CLI
uv run gorgonetics db-snapshots
```

Programmatically, the database class exposes:

- `get_snapshots()` -- returns all snapshot IDs, timestamps, and schema versions.
- `get_table_changes(table, start_snapshot, end_snapshot)` -- returns row-level diffs between two snapshots.
- `cleanup_old_files(dry_run=True)` -- removes orphaned Parquet files from old snapshots.

## Query Patterns

### Named parameters

The codebase uses named parameters exclusively:

- **DuckLake** (DuckDB): `$param` style — `SELECT * FROM pets WHERE id = $pet_id`
- **Auth database** (SQLite): `:param` style — `SELECT * FROM users WHERE id = :id`

Positional placeholders (`?`) are not used per project convention.

### Pagination

`get_all_pets()` supports `limit` and `offset` parameters for paginated queries:

```python
pets = db.get_all_pets(user_id=1, species="beewasp", limit=20, offset=0)
total = db.count_pets(user_id=1, species="beewasp")
```

### Duplicate detection

Pets are deduplicated by `content_hash` (SHA-256 of the original genome file):

```python
existing = db.find_pet_by_hash(content_hash)
if existing:
    # Pet already uploaded
```

## Backup and Restore

### Backup (Docker container)

```bash
# Back up from a running container
./scripts/db-backup.sh <container_name> <backup_dir>

# Example
./scripts/db-backup.sh gorgonetics ./backups
```

The script runs `CHECKPOINT` inside the container to flush the WAL, then archives the `/app/data` directory into a timestamped tarball.

### Restore (Docker volume)

```bash
# Restore into a Docker volume (container must be stopped)
./scripts/db-restore.sh <backup_file> [volume_name]

# Example
./scripts/db-restore.sh ./backups/gorgonetics_20260311_120000.tar.gz gorgonetics_data
```

### Local backup

For local development, copy the auth database, catalog, and data directory:

```bash
cp users.sqlite users.sqlite.bak
cp metadata.sqlite metadata.sqlite.bak
cp -r data/ data_backup/
```

## Troubleshooting

### Database not initializing

Check that the DuckLake extension is installable. DuckDB downloads extensions on first use, so network access is required the first time:

```bash
uv run python -c "import duckdb; c = duckdb.connect(); c.execute('INSTALL ducklake'); print('OK')"
```

### "Table does not exist" errors

The database auto-creates tables on connection. If you see catalog errors, the DuckLake attachment may have failed. Run `uv run gorgonetics db-status` to verify configuration and check that `metadata.sqlite` and the `data/` directory are accessible.

### Stale Parquet files

Over time, DuckLake accumulates old Parquet files from previous snapshots. Run cleanup periodically:

```bash
uv run gorgonetics db-cleanup              # dry run first
uv run gorgonetics db-cleanup --no-dry-run # then delete
```

### Resetting the database

Delete both databases and the data directory, then re-populate:

```bash
rm -f users.sqlite metadata.sqlite
rm -rf data/
uv run gorgonetics populate
uv run gorgonetics create-admin --username admin --password yourpassword
```
